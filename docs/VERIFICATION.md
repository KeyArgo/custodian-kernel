# Verification

This document explains exactly how a skeptical third party can verify every
real claim about Custodian — without needing to trust the author's word.

## 1. A real Stripe PaymentIntent is on record

The PaymentIntent ID `pi_3TkZWEPfSF4TGXT90AWlrnle` appears in the test
suite (`tests/test_ledger.py`, `tests/test_audit.py`, `tests/test_types.py`
and their test assertions) as a real value used during the demo.

**Important correction**: Stripe objects are scoped to the account that created
them. Your own Stripe test-mode key — no matter whose it is — cannot retrieve
this PaymentIntent; Stripe returns "no such payment_intent" for any account
other than the one that created it. That's a real Stripe platform boundary,
not a limitation of this project, but it does mean "run this with your own
key" (an earlier, inaccurate version of this doc) doesn't actually work as a
self-service check.

What actually verifies it, in order of strength:

1. **Watch it happen live.** Run a real spend through the system and watch the
   PaymentIntent appear in the project's own Stripe test dashboard in the same
   moment. The timing has to line up in real time, which is hard to fake.
2. **Ask for restricted, view-only access** to the project's actual Stripe
   test-mode dashboard (Stripe supports inviting team members with a
   read-only role) — this lets a real reviewer look at the real account
   directly, not just take an ID's existence on faith.
3. **Run `verify_kit.py`** (this repo's root) — it re-runs the test suite and
   the self-approval regression proof live, and pulls fresh data from the
   real public dashboard. It does not attempt to verify the Stripe object
   directly, for the reason above.

The ID itself (`pi_3TkZWEPfSF4TGXT90AWlrnle`) is real and was generated by a
real Stripe API call during the sandbox demo — it's just not something a
third party can independently query without access to the account that
created it.

## 4. The self-approval regression test

The file `tests/test_self_approval_regression.py` is the most important
test for the security claim. It contains 7 tests:

- **`test_spend_v2_has_no_approved_by_flag`** — statically parses
  `skills/payments/stripe-spend/scripts/spend_v2.py` and asserts
  `approved_by` is not in its argument list. If anyone ever adds an
  `--approved-by` flag back to the unprivileged request path, this test
  fails.
- **`test_spend_v2_only_has_expected_flags`** — pins the exact allowed
  flag set for `spend_v2.py`. If a new flag is added under a different
  name (e.g. `--override`, `--force-approve`), this test catches it.
- **`test_approve_v2_requires_approved_by_and_a_code`** — confirms the
  privileged path requires both a human name AND a code.
- **`test_approve_v2_calls_check_response_before_executing`** — confirms
  verification happens before execution, not after.
- **`test_check_response_is_abstract_and_must_hit_an_external_source`** —
  confirms `ApprovalBackend.check_response` is abstract.
- **`test_twilio_backend_check_response_makes_an_http_call`** — confirms
  the real backend hits `verify.twilio.com` via HTTP, not local state.
- **`test_pending_approval_never_stores_the_code`** — confirms
  `PendingApproval` dataclass has no code field.

The fix this test protects was proven by deliberately reintroducing the
bug (adding `--approved-by` back), confirming the test failed, then
removing it. You can reproduce this:

```bash
# Edit skills/payments/stripe-spend/scripts/spend_v2.py to add:
# parser.add_argument("--approved-by")
# Then run:
pytest tests/test_self_approval_regression.py -v
# Expected: test_spend_v2_has_no_approved_by_flag FAILS

# Now revert the edit and confirm it passes again.
```

## 5. Real Twilio Verify integration

The file `custodian/backends/twilio_verify.py` is a real Twilio Verify
integration. The methods `send_challenge()` and `check_response()` make

The test in `test_backends.py` validates the secret-file loading logic
without making real API calls, but the code itself is production-grade
`requests` calls to Twilio's API.

## 6. The kill switch actually stops the real, live script

This is reproducible against the real sandbox, not a parallel test path.
`spend.py` runs inside the NemoClaw sandbox; `custodian kill`/`custodian
resume` run on the host, where the `custodian` package is actually
installed (it's deliberately *not* installed inside the sandbox -- see
`docs/ARCHITECTURE.md`'s kill switch section). Both sides point at the same
`state/` directory, shared bidirectionally between host and sandbox. The
sequence that was actually run, with real Stripe PaymentIntents on both
sides of the kill:

```bash
# Inside the sandbox: a real autonomous spend succeeds
python3 /sandbox/.hermes/skills/payments/stripe-spend/scripts/spend.py \
  --amount 0.50 --description "before kill"
# -> [stripe] PaymentIntent created: pi_3Tl1bqPfSF4TGXT90AWlrnle

# On the host: engage the real kill switch, pointed at the shared state dir
custodian kill --by Operator --reason "demonstration" \
  --state-dir <host-mounted-path>/stripe-spend/state

# Inside the sandbox: the exact same request, now denied by the real script
python3 /sandbox/.hermes/skills/payments/stripe-spend/scripts/spend.py \
  --amount 0.50 --description "after kill"
# -> [authority] DENIED -- kill switch is engaged ... exit code 3

# On the host: release it
custodian resume --by Operator --state-dir <host-mounted-path>/stripe-spend/state

# Inside the sandbox: real spend succeeds again
python3 /sandbox/.hermes/skills/payments/stripe-spend/scripts/spend.py \
  --amount 0.50 --description "after resume"
# -> [stripe] PaymentIntent created: pi_3Tl1cRPfSF4TGXT91Bs7289S
```

The real `audit_log.jsonl` in that state directory shows the complete,
unedited sequence: `executed` → `denied` → `executed`. Without sandbox
access you can reproduce the same logic locally with `custodian kill`/
`custodian request`/`custodian resume` against any workspace created by
`custodian init` -- see `docs/GETTING_STARTED.md` step 9 for the exact
commands and expected output.

## 7. Confirm the CLI surface matches the documentation

The CLI is defined in `custodian/cli/main.py`. Every documented command
and flag corresponds to an `add_argument` call in that file:

- `custodian init --dir`
- `custodian validate <policy_path>`
- `custodian status --state-dir`
- `custodian audit --state-dir --limit --event`
- `custodian request --amount --description --skill --context --state-dir --policy`
- `custodian approve <code> --approved-by --state-dir`
- `custodian deny --denied-by --state-dir`
- `custodian kill --by --reason --state-dir`
- `custodian resume --by --state-dir`

No undocumented commands or flags exist in the code. No documented
commands or flags are absent from the code.
